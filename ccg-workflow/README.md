# CCG Workflow - OpenSkill 版本

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](https://opensource.org/licenses/MIT)

一个轻量级、命令驱动的多智能体协调技能，专为软件开发设计。该技能协调多个 AI 模型（Claude、Codex、Gemini）来协助功能开发、调试和代码审查等任务，所有操作都通过 AI 助手内简单的斜杠命令界面完成。

---

## 🚀 快速开始

### 1. 前置条件

该技能充当协调的"大脑"。它需要底层 CLI 工具才能运行。请确保以下工具已安装并在 `PATH` 中可用：

*   **AI 助手**：能够加载和执行 OpenSkills 的兼容 AI 助手（例如 Manus、Claude Code）。
*   **`codeagent-wrapper`**：用于调用不同模型后端的统一 CLI。
*   **模型 CLI**：`codex`、`gemini` 和 `claude` 的命令行工具。

### 2. 安装

#### 方式一：使用安装脚本（推荐）

```bash
# 克隆或下载技能包
git clone <repository-url> ccg-workflow
cd ccg-workflow

# 运行安装脚本
./install.sh
```

安装脚本会将斜杠命令链接到 `~/.claude/commands/ccg/` 目录，使您可以在任何项目中使用 `/ccg:propose`、`/ccg:explore` 等命令。

#### 方式二：手动安装

1.  下载最新的 `ccg-workflow-openskill-*.zip` 安装包。
2.  将安装包解压到您 AI 助手的技能目录（例如 `~/.config/manus/skills/` 或 `~/.claude/skills/`）。
3.  手动创建命令链接：
    ```bash
    mkdir -p ~/.claude/commands/ccg
    ln -s /path/to/ccg-workflow/commands/*.md ~/.claude/commands/ccg/
    ```

#### 卸载

如需卸载，运行：

```bash
./uninstall.sh
```

### 3. 首次运行

要熟悉工作流程，请启动交互式入门教程：

```
/ccg:onboard
```

## 💬 命令参考

### `/ccg:propose <描述>`

**从想法到计划的快速路径。** 这是最常用的起点。它接受功能或任务的自然语言描述，并生成一个完整的、经过多模型验证的实现计划供您审批。

*   **用法**：`/ccg:propose <您想构建的内容>`
*   **参数**：
    *   `<描述>`（必需）：对所需变更的清晰一句话描述。
*   **适用场景**：适用于需求明确、定义清晰的小型到中型功能。它自动化了分析、规划和构思阶段。
*   **示例**：
    ```
    /ccg:propose 为文档中的代码块添加"复制到剪贴板"按钮。
    ```

### `/ccg:explore <主题>`

**多模型头脑风暴会议。** 使用此命令来调查技术问题、比较方法或思考问题，而不会创建任何永久性文件或产物。这是一种低摩擦的方式来利用后端和前端专家模型的不同视角。

*   **用法**：`/ccg:explore <主题或问题>`
*   **参数**：
    *   `<主题>`（必需）：您想调查的问题或主题。
*   **适用场景**：非常适合早期构思、架构决策，或当您不确定最佳前进方向时。
*   **示例**：
    ```
    /ccg:explore 我们应该使用 WebSocket 还是 Server-Sent Events 来实现实时通知？
    ```

### `/ccg:new <变更名称>`

**启动一个细粒度的、受控的变更。** 此命令启动扩展工作流程，为您提供逐步控制。它为变更创建一个专用目录并设置依赖图，但在您使用 `/ccg:continue` 或 `/ccg:ff` 继续之前不会创建任何产物。

*   **用法**：`/ccg:new <变更名称>`
*   **参数**：
    *   `<变更名称>`（必需）：变更的简短描述性名称（例如 `refactor-auth-service`）。
*   **适用场景**：用于大型、复杂的功能，或当您想手动检查规划过程的每个阶段时。
*   **示例**：
    ```
    /ccg:new feature/user-profiles
    ```

### `/ccg:continue [变更名称]`

**执行下一步。** 在使用 `/ccg:new` 后，此命令创建处于 `READY` 状态的下一个单个产物。它允许您一步一步地完成工作流程。

*   **用法**：`/ccg:continue`
*   **参数**：
    *   `[变更名称]`（可选）：要继续的变更名称。默认为活动变更。
*   **适用场景**：非常适合探索性工作流程，您希望在进入下一阶段之前审查每个阶段（例如 `research`、`ideation`）的输出。
*   **示例**：
    ```
    /ccg:continue
    ```

### `/ccg:ff [变更名称]`

**快进到计划。** 此命令自动按顺序创建所有规划相关的产物（`enhance`、`research`、`ideation`、`plan`），仅在需要用户批准时停止。

*   **用法**：`/ccg:ff`
*   **参数**：
    *   `[变更名称]`（可选）：要快进的变更名称。默认为活动变更。
*   **适用场景**：两全其美。它提供了扩展工作流程的结构，但具有 `/ccg:propose` 的速度。当您已使用 `/ccg:new` 开始并对方向有信心时使用它。
*   **示例**：
    ```
    /ccg:ff
    ```

### `/ccg:apply [变更名称]`

**实施计划。** 这是代码编写阶段。协调器执行已批准计划中的任务，向专家模型咨询建议，但保留对文件系统的唯一写入权限。

*   **用法**：`/ccg:apply`
*   **参数**：
    *   `[变更名称]`（可选）：要应用的变更名称。默认为活动变更。
*   **适用场景**：在通过 `/ccg:propose` 或 `/ccg:ff` 创建并批准计划后运行此命令。
*   **示例**：
    ```
    /ccg:apply
    ```

### `/ccg:verify [变更名称]`

**多模型代码审查。** 实施后，此命令启动并行代码审查。后端和前端模型都检查变更，他们的发现会被评分和过滤，以提供高信号报告。

*   **用法**：`/ccg:verify`
*   **参数**：
    *   `[变更名称]`（可选）：要验证的变更名称。默认为活动变更。
*   **适用场景**：在 `/ccg:apply` 之后运行的基本质量关卡。它有助于在提交之前捕获错误、安全漏洞和偏离最佳实践的问题。
*   **示例**：
    ```
    /ccg:verify
    ```

### `/ccg:archive [变更名称]`

**完成并提交。** 此命令结束工作流程。它根据完成的工作生成规范的提交消息，归档所有规划和执行产物，并清理工作目录。

*   **用法**：`/ccg:archive`
*   **参数**：
    *   `[变更名称]`（可选）：要归档的变更名称。默认为活动变更。
*   **适用场景**：工作流程的最后一步，在所有实施和验证完成后运行。
*   **示例**：
    ```
    /ccg:archive
    ```

### `/ccg:status [变更名称]`

**查看工作流程仪表板。** 此命令提供变更当前状态的摘要，显示哪些产物处于 `BLOCKED`、`READY` 或 `DONE` 状态。

*   **用法**：`/ccg:status`
*   **参数**：
    *   `[变更名称]`（可选）：要查看的变更名称。如果省略，显示所有活动变更。
*   **适用场景**：在任何时候都很有用，用于了解您在工作流程中的位置并查看下一步是什么。
*   **示例**：
    ```
    /ccg:status feature/user-profiles
    ```

### `/ccg:config <操作>`

**管理配置。** 此命令允许您查看和修改技能的设置。

*   **用法**：`/ccg:config <show|init|profile|schema>`
*   **参数**：
    *   `show`：显示当前配置。
    *   `init`：在项目根目录创建 `config.yaml` 文件。
    *   `profile`：在命令配置文件之间切换（例如 `core`、`expanded`）。
    *   `schema`：管理工作流程模式（例如 `list`、`fork`、`use`）。
*   **适用场景**：在新项目上使用 `init` 进行设置。使用 `show` 调试行为。使用 `schema fork` 创建自定义工作流程。
*   **示例**：
    ```
    /ccg:config init
    /ccg:config schema fork default my-fast-workflow
    ```

### `/ccg:onboard`

**交互式教程。** 此命令提供整个工作流程的引导式演示，解释每个步骤和命令。

*   **用法**：`/ccg:onboard`
*   **适用场景**：任何新用户的最佳起点。
